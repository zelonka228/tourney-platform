# Тиждень 4 (продовження) — реєстрація користувачів, ролі, особистий кабінет

Специфікація для розширення наявної JWT-авторизації (`backend/src/auth.js`, два фіксовані
сід-акаунти Admin/User без самореєстрації) до повноцінної системи акаунтів із
самореєстрацією, трьома ролями, особистим кабінетом і адмінською панеллю керування
користувачами. Узгоджено з користувачем через brainstorming-скіл (2026-07-16).

## Ролі

Замість поточних двох (`admin`/`user`) — три:

- **`admin`** — повний доступ: усе, що `organizer` (нижче), плюс керування акаунтами
  (`/admin/users`) і ручна зміна рідкості картки команди (`rarityOverride`).
- **`organizer`** — новий рівень. Той самий обсяг прав, що зараз має `admin` для команд,
  турнірів і матчів: створення/редагування/видалення команди, створення/редагування/
  видалення турніру, реєстрація команд у турнір, генерація сітки, ручне розставлення
  посіву, введення рахунку матчу, скасування результату матчу. **Не має** доступу до
  керування акаунтами і до `rarityOverride`.
- **`user`** — без змін: лише перегляд (команди, турніри, сітки, статистика гравців),
  завантаження PNG-картки команди (публічна дія, авторизація не потрібна взагалі).

Самореєстрація завжди створює `role: "user"`. Підвищити до `organizer`/`admin` може
лише `admin` через `/admin/users`.

## Дані: зміни в Prisma-схемі

```prisma
model User {
  id            Int      @id @default(autoincrement())
  username      String   @unique
  usernameLower String   @unique   // нове — регістронезалежна унікальність (SQLite
                                    // не підтримує mode:"insensitive" у Prisma, тож
                                    // унікальність гарантує сама БД через цю колонку)
  passwordHash  String
  role          String   // "admin" | "organizer" | "user"
  bio           String?  // нове, до 500 символів
  avatar        String?  // нове, base64 dataURL — той самий підхід що й Team.logo
  createdAt     DateTime @default(now())
}

model Team {
  ...
  rarityOverride String?  // нове — "Common" | "Rare" | "Epic" | "Legendary" | null (null = авто)
}
```

`usernameLower` завжди `username.toLowerCase()`, підтримується синхронно при створенні й
при зміні логіна адміном.

## Пароль — стандарт складності

Валідація однакова на фронтенді (для миттєвого фідбеку) і на бекенді (джерело правди).

**Жорсткі вимоги (блокують реєстрацію й зміну пароля адміном):**
- довжина 8–72 символи (72 — межа bcrypt);
- містить хоча б одну велику літеру, одну малу, одну цифру;
- не збігається (без урахування регістру) з логіном.

**Рекомендації (підказки під полем, не блокують):**
- 12+ символів;
- додати спецсимвол (`!@#$%^&*` тощо);
- уникати словникових слів і особистої інформації.

На формі реєстрації — індикатор складності (слабкий/середній/сильний/дуже сильний),
що рахується на льоту з довжини й різноманітності символів, і статичний список
рекомендацій під полем. Той самий індикатор і той самий текстовий інпут для нового
пароля використовується в адмінській формі "скинути пароль".

Чорний список найпоширеніших паролів — **свідомо не реалізується** (по прямому запиту
користувача, спрощення обсягу).

## Реєстрація

`POST /api/auth/register` (публічний роут):
- тіло: `{ username, password }`;
- логін: 3–20 символів, літери/цифри/підкреслення;
- пароль — за правилами вище;
- перевірка унікальності через `usernameLower`, конфлікт → `409`;
- порушення правил пароля/логіна → `400` з конкретним повідомленням;
- успіх: створює `User` з `role: "user"`, одразу логінить (та сама відповідь, що й
  `POST /api/auth/login`: `{ token, user }`).

Форма реєстрації — нова сторінка `/register` (лінк з `/login`), у стилі наявного `Login.jsx`.

## Особистий кабінет (`/account`)

- Власник бачить і редагує **своє** біо (textarea, ліміт 500 символів) і аватарку.
- Аватарка — той самий crop-to-square patern, що вже є для `Team.logo` в `Team.jsx`
  (canvas-кроп по центру 256×256 → JPEG dataURL, ліміт ~300KB). Логіка кропу
  виноситься в спільний `frontend/src/lib/cropImage.js`, щоб не дублювати між
  `Team.jsx` і новою формою кабінету.
- `PUT /api/auth/me` (потрібна авторизація) — приймає лише `{ bio, avatar }`; логін і
  роль користувач сам не змінює.
- Дані приватні: у відповіді API біо/аватарку іншого користувача бачить тільки він сам
  і admin. Публічних сторінок профілю немає.
- Бейдж ролі на сторінці (як зараз, але три варіанти замість двох: admin/organizer/user
  — різний колір/підпис).

## Адмінська панель керування акаунтами

Нова сторінка `/admin/users`, доступна лише `role === "admin"` (лінк з `/account`,
поза головною навігацією).

- **Список**: логін, роль, дата реєстрації, наявність біо (індикатор), дії.
- **Редагування** (`PUT /api/admin/users/:id`, admin-only): логін, роль (select
  admin/organizer/user), біо, аватарка.
- **Скид пароля** (`POST /api/admin/users/:id/reset-password`, admin-only): адмін вводить
  новий пароль (той самий валідатор складності + індикатор), підтверджує — бекенд хешує
  й замінює `passwordHash`.
- **Видалення** (`DELETE /api/admin/users/:id`, admin-only): підтвердження через уже
  наявний у проєкті патерн confirm-панелі. Заборонено видаляти:
  - самого себе через цю панель (захист від випадкового самовидалення);
  - останнього залишкового `admin` в системі (захист від блокування без адмінів).
- Усі відповіді `/admin/users*` ніколи не містять `passwordHash`.
- У User немає жодних FK-звʼязків із Team/Player (акаунт повністю окремий від складу
  команди, підтверджено раніше в brainstorming) — видалення користувача не має
  каскадних наслідків для команд/турнірів.

## Права `organizer` на бекенді

Новий middleware `requireContentManager` (`backend/src/auth.js`):

```js
export function requireContentManager(req, _res, next) {
  if (!req.user) throw new HttpError(401, "Потрібна авторизація.");
  if (req.user.role !== "admin" && req.user.role !== "organizer") {
    throw new HttpError(403, "Потрібні права організатора або адміністратора.");
  }
  next();
}
```

Замінює `requireAdmin` на:
- `POST/PUT/DELETE /api/teams`;
- `POST/PUT/DELETE /api/tournaments`, `/register`, `/generate-bracket`,
  `/teams/:teamId` (видалення команди з чернетки), `/teams/reorder`;
- `PUT /api/matches/:id/score`, `POST /api/matches/:id/reset`.

`requireAdmin` (без змін, строго `role === "admin"`) лишається на:
- усіх `/api/admin/users*`;
- полі `rarityOverride` всередині `PUT /api/teams/:id` — це єдиний роут, де два рівні
  прав перетинаються: `organizer` може редагувати команду (назва, лого, склад тощо), але
  якщо в тілі запиту присутнє `rarityOverride`, роут окремо перевіряє
  `req.user.role === "admin"` і кидає `403`, якщо ні — незалежно від того, що
  `requireContentManager` уже пропустив запит далі.

JWT-пейлоад (`signToken`) без змін у структурі — `role` вже там; нового поля не додається,
бо всі перевірки й далі йдуть через `role`.

## Права на фронтенді

`frontend/src/lib/auth.jsx`, `AuthProvider` віддає:
- `isAdmin` — `user?.role === "admin"` (як зараз, без змін значення);
- `canManageContent` — `role === "admin" || role === "organizer"`.

Гейтинг сторінок/кнопок переводиться з `isAdmin` на `canManageContent` скрізь, де мова
про команди/турніри/матчі:
- `Create.jsx` (створення турніру) — гейт і сторінка-заглушка "потрібні права";
- `Team.jsx` (створення/редагування команди) — гейт і заглушка;
- `Tournament.jsx` — усі місця, де зараз `isAdmin` керує кнопками видалення турніру,
  генерації сітки, реєстрації, реордера посіву, відкриття модалки рахунку/скасування.

`isAdmin` (без зміни семантики) лишається джерелом гейту для:
- пункту "Рідкість (вручну)" у формі `Team.jsx` — рендериться тільки для `isAdmin`;
- лінку на `/admin/users` зі сторінки `/account`;
- самого маршруту `/admin/users`.

## Ручна зміна рідкості картки

- У формі `/team/:id` (видима тільки коли `isAdmin`) — новий select "Рідкість (вручну)":
  `Авто (обчислюється)` / `Common` / `Rare` / `Epic` / `Legendary`.
- `PUT /api/teams/:id` приймає `rarityOverride`; порожнє значення/`"Авто"` → `null` у БД;
  інакше валідується проти `["Common","Rare","Epic","Legendary"]` (`requireEnum`).
- `teamRarity(team)` у `frontend/src/lib/demo.js` перевіряє `team.rarityOverride` першим
  рядком і повертає його без змін, якщо задано — інакше стара логіка на основі `best`/
  `tournaments`, без змін. Це один рядок на початку функції — впливає одразу на всі три
  місця виклику (`TeamCard.jsx`, `Profile.jsx` панель рідкості, `Profile.jsx` список).

## Навігація

`App.jsx`: пункт `nav.create` (`/create`) рендериться лише коли `canManageContent`
(зараз видимий усім незалежно від ролі й статусу авторизації, хоча сторінка вже
блокує non-admin). Приховування пункту навігації закриває сам запит користувача —
"прибрати кнопку створити турнір у неавторизованих і звичайних користувачів".

## Безпека — свідомо поза межами цієї ітерації

Узгоджено з користувачем як прийнятні спрощення для навчального проєкту:
- чорний список типових паролів;
- rate-limiting спроб логіну/реєстрації;
- email-підтвердження, відновлення пароля через пошту, 2FA;
- самостійна зміна пароля користувачем (лишається тільки примусовий скид адміном);
- інвалідація вже виданих JWT при зміні ролі — застосовується тільки після наступного
  логіну/спливання токена (12 год), як і зараз для будь-якої зміни ролі.

## Тестовий план

- `backend/test-api.js`: нові сценарії — реєстрація (успіх, дубль логіна регістронезалежно,
  слабкий пароль → 400 з конкретних причин), `organizer`-акаунт створює команду/турнір/
  проводить матч без прав `admin`, `organizer` отримує 403 на `/admin/users*` і на
  `rarityOverride`, `admin`-only видалення користувача блокується для себе/останнього admin.
- Ручна перевірка в браузері: реєстрація нового акаунта → одразу залогінений як `user` →
  кнопка "Створити" відсутня в навігації → адмін підвищує його до `organizer` через
  `/admin/users` → після повторного логіну бачить кнопку "Створити" і може провести
  турнір повністю (створення → реєстрація команд → сітка → рахунок) без прав `admin`;
  окремо перевірити, що `organizer` не бачить `/admin/users` і не бачить поля "Рідкість
  (вручну)".
