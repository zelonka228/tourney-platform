# Подвійне вибування (double elimination) — специфікація

Поле `Tournament.bracketType: "single" | "double"` існує в схемі й у формі створення
турніру з самого початку, але `"double"` досі блокується (400 на бекенді, форма не
дає надіслати на фронті). Цей документ описує, що потрібно, щоб розблокувати його.

## Обсяг (scope)

- **Тільки степені двійки** (4/8/16/32 команди) для першої версії. Довільна
  кількість команд із баями в сітці невдах — суттєво складніший алгоритм
  (потрібно коректно розсіювати баї так, щоб не було передчасних повторних
  зустрічей), відкладено на потім.
- **Гранд-фінал — один матч, без bracket reset.** Спершу було реалізовано
  "справжнє" double elimination з реваншем (якщо LB-фіналіст виграє перший
  матч гранд-фіналу — грається другий, вирішальний), як на Challonge/Toornament.
  Користувач, побачивши це вживу, попросив спростити: WB-чемпіон грає з
  LB-чемпіоном один матч, переможець — чемпіон турніру, без другого шансу для
  жодної зі сторін.

## Модель даних

`Match` отримує нове поле `bracket: "winners" | "losers" | "final"`
(для `bracketType: "single"` завжди `"winners"` — існуючі турніри й дані міграції
не потребують). Раунди нумеруються окремо всередині кожного бракету:

- `winners`: `0 .. log2(n)-1`, як зараз.
- `losers`: `0 .. 2*(log2(n)-1)-2`.
- `final`: один рядок, `round 0` — єдиний матч гранд-фіналу.

## Побудова сітки

Winners bracket будується як і зараз (`buildBracketRows`, без змін). Losers
bracket будується новою функцією за стандартним алгоритмом double-elimination:
кожен LB-раунд по черзі приймає або програних з чергового WB-раунду, або
переможців попереднього LB-раунду; порядок посіву всередині LB навмисно
переставлений так, щоб команди з однієї половини winners-сітки не зустрілися в
losers занадто рано. Це єдина по-справжньому нетривіальна частина коду —
винести в окрему, добре покриту тестами функцію в `backend/src/bracket.js`
(не переписуючи існуючу логіку для `single`).

## Просування переможця/переможеного

`backend/src/advance.js` отримує другу функцію поруч із `advanceWinner`:

- `advanceWinner` — як зараз, переможець їде далі у своєму бракеті
  (для `winners`-матчу — у наступний `winners`-раунд; для `losers`-матчу — у
  наступний `losers`-раунд; для `final` — нікуди, це завжди термінальний матч).
- `advanceLoser` (новий) — програний з `winners`-матчу їде у відповідний слот
  `losers`-сітки; програний з `losers`-матчу вибуває з турніру остаточно.

Гранд-фінал: `final round 0` — WB-чемпіон проти LB-чемпіона, один матч.
Хто б не переміг — турнір одразу `completed`.

## API

- `POST /api/tournaments` і `POST /api/tournaments/:id/generate-bracket`:
  прибрати безумовний 400 для `bracketType: "double"`; замість цього 400 лише
  коли кількість команд не степінь двійки (з чітким повідомленням про причину).
- `PUT /api/matches/:id/score`: для матчу з `bracket: "winners"` після
  `advanceWinner` додатково викликати `advanceLoser`.
- `POST /api/matches/:id/reset`: має вміти відкотити і loser-дроп (не лише
  winner-просування), з тим самим захистом "якщо наступний матч уже зіграно —
  спершу скасуйте його результат" — тепер це стосується і того матчу, куди
  поїхав переможець, і того, куди поїхав переможений.

## Фронтенд

- `Create.jsx`: опція `"double"` розблоковується, залишаючись задизейбленою
  (з уточненим текстом підказки) коли кількість вибраних команд не степінь
  двійки.
- `Tournament.jsx`, вкладка "Сітка": стек-розкладка — winners bracket
  рендериться як зараз (той самий `MatchCard`/SVG-провода), під ним окремим
  блоком losers bracket тим самим компонентом, праворуч — колонка гранд-фіналу
  з одним матчем. Банер чемпіона й конфеті тригеряться від результату цього
  єдиного матчу.
- Вкладка "Результати": додається позначка бракету (winners/losers/final) біля
  кожного рядка.
- Нові підписи (укр/eng): "Сітка переможців" / winners bracket, "Сітка невдах"
  / losers bracket, "Гранд-фінал" / grand final.

## Тестування

`backend/test-api.js` доповнюється сценаріями для 4-командного double
elimination: повний прогін турніру з перемогою WB-чемпіона у фіналі, повний
прогін з перемогою LB-чемпіона у фіналі (в обох випадках турнір завершується
одразу після цього єдиного матчу), та reset (скасування) матчу з `winners`,
який уже встиг скинути програного в `losers`.
